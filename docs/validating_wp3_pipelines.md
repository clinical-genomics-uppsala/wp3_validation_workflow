# Running the wp3 validation pipeline

## Creating the validation checksum file for the first time

1. Change to the results folder of the pipeline 
2. Create a tsv file listing all the results files to be checksumed. This file lists the files to be checksumed in the file column and the checksum column can be left blank.
3. Save this file under /projects/wp3/Validering/pipelines/pipeline_md5sums and add the path to the md5sum file in the pipeline specific config under ```input:```.

Below is an example of creating the checksums for the Nallo wgs pipeline.

```bash
snakemake create_validation_data --profile /projects/wp3/Validering/pipelines/wp3_validation_workflow/profiles/slurm  --configfiles /projects/wp3/Validering/pipelines/wp3_validation_workflow/config/config.yaml /projects/wp3/Validering/pipelines/wp3_validation_workflow/config/config_nallo_hifi_wgs.yaml -s /projects/wp3/Validering/pipelines/wp3_validation_workflow/workflow/Snakefile 
```

3. Copy the newly generated checksum tsv file to the pipeline_md5sums folder with the specific name specified in the config for that pipline (in this example config/config_nallo_hifi_wgs.yaml).

```bash
cp validation_results/new_validation_data.tsv /projects/wp3/Validering/pipelines/pipeline_md5sums/nallo_hifi_wgs_md5sums.tsv
```

## Validating the results from a new version of a piepline

When updatng to a newer version of a pipeline, or updating the config for a pipeline, the validation pipeline can be used to see which files have changed. This will automatically run the hap.py and truvari benchmarking analysis for any HG001 and HG002 samples found in the results.

```bash
snakemake  --profile /projects/wp3/Validering/pipelines/wp3_validation_workflow/profiles/slurm  --configfiles /projects/wp3/Validering/pipelines/wp3_validation_workflow/config/config.yaml  /projects/wp3/Validering/pipelines/wp3_validation_workflow/config/config_nallo_hifi_wgs.yaml -s /projects/wp3/Validering/pipelines/wp3_validation_workflow/workflow/Snakefile 
```

Look for any unexpected changes. In some cases, like an update to a caller, annotation source or filter, we would expect the md5sum to change in a vcf. 


## Gotchas.
### Files not in the md5sum tsv foile

In some cases an updated pipeline will produce new files that are not in the md5sum tsv file. In that case update the tsv file with the name of the files and it md5sum as generated by the  pipeline. The pipline will produce a log file of all the missing files that lists there md5sum as calculated by the pipeline.


## Investigating what has caused the change in a vcf


### Comparing SNV and Indel vcf files

```bash
bcftools isec -p HG24385_hard-filtered_isec v0.8.1_config_v1.0.4/snvs/family/HM24385/HM24385.hard-filtered.vcf.gz v0.11.0_config_v1.3.0/snvs/family/HM24385/HM24385.hard-filtered.vcf.gz

```

Check the folder with the vcf results

```bash
cd  HG24385_hard-filtered_isec
grep -v ^'#' -c *.vcf
```

In the bcftools isec results folder files 0000.vcf and 0001.vcf list variants private to each vcf. The grep count above should return zero if the variants in each vcf compared are the same.

### Highlighting changes with git

If the bcftools isec shows zero differences between the vcfs, then you can use git diff to highlight what has changed in the vcf rows to cause the difference in md5 checksum. This is a good way
to highlight annoation differences in the info field. For instance, if an additional frequency database was used that newly added annotation source will be highlighted.

```bash
git diff --word-diff --word-diff-regex=.  <(zcat v0.8.1_config_v1.0.4/snvs/family/HM24385/HM24385.hard-filtered.vcf.gz) <(zcat v0.11.0_config_v1.3.0/snvs/family/HM24385/HM24385.hard-filtered.vcf.gz)
```
